播放音檔時「逐字 highlight」

有哪些實踐方式 · 現有架構好不好做 · 建議怎麼做 — 配套技術 spec：spec.md

一句話結論：這是加法（additive），不是重構。現有的 hash + manifest + new Audio() 主幹原封不動。關鍵設計是把「逐詞時間來源」藏在一個穩定介面 （<hash>.words.json sidecar） 後面，這樣未來把語音換成 GPT-4o（不吐時間）也不會把逐字 highlight 弄壞。

① 先看效果：精準 vs. 會漂移

同一句、同一個播放時鐘，上排用真實逐詞時間（Edge WordBoundary）， 下排用均分估算（總長 ÷ 詞數，每個詞給一樣的時間）。按播放，看下排怎麼在短詞（on、a）一帶開始對不上聲音——那個落差就是「漂移」。

④ 建議架構：sidecar 當「解耦介面」

解耦的縫，是一份穩定、內容定址的逐詞時間檔：runtime 只認它，多個 producer 都能寫它。

為什麼這個介面對：

  • 白搭現有設計：內容定址 hash 本來就把 text→MP3，sidecar 共用同一個 hash，快取/idempotency 全免費。

⑥ 兩個會「靜默壞掉」的雷（怎麼來、怎麼避免）

先懂背景：app 不是「一段文字配寫死的檔名」，而是內容定址—— 段落文字 → 正規化 → SHA-256 前 12 碼 → 那個 hash 就是 MP3 檔名。 build 時算一次 hash 命名 MP3，runtime 再從畫面那段文字重算一次去查表。 兩次必須算出完全一樣的 hash 才找得到音檔。懂這個，兩個雷就清楚了。

🔴 雷 1 — Hash drift（hash 對不上 · 最高嚴重度）

面向說明
是什麼包詞成 <span> 時若改到了那段文字，runtime 重算的 hash 跟 MP3 檔名對不上 → 查表 miss → 每段都靜默退回瀏覽器機器人語音。不報錯，整站發音悄悄變爛。
為什麼會中<span>foo</span> 的 textContent 本來等於 foo，雷在細節：切詞時多塞/吃掉空白（fresh␣␣coffee 兩格被收一格）、把要剝除的講者 <strong> 標籤也算進去、或彎引號 '↔直引號 ' 沒對齊。
怎麼避免① 切詞器只「切段＋包 span」，逐字元含空白原封不動。② build 時加斷言：包前 textContent === 包後串回（或包完重算 hash、確認仍在 manifest）→ 漂移就讓 build 失敗。③ 講者標籤剝除跟 speech.ts 用同一份程式碼、不另寫。

精髓：把「runtime 才會發現的靜默失敗」提前變成「build 時的大聲報錯」——漂移上不了線。便宜、可完全消除。

🟠 雷 2 — Tokenization drift（兩邊的「詞」對不齊 · 已查證）

面向說明
是什麼有兩份「詞清單」：畫面 DOM span（按空白切）與 Edge WordBoundary（每詞帶時間）。兩份有時對不起來。
為什麼會中Edge 念之前改寫 token：100→one hundred（一變二）、USD→United States Dollars（一變三），且只給「念的詞＋時間」、不給輸入字元位置。天真假設「第 i 詞=第 i 詞」，一碰到 100 就從此往後錯一格、錯到段尾，同樣靜默、只是亮錯字。
怎麼避免① 不假設 1:1，用序列對齊（雙指標 / Needleman-Wunsch）；一個 DOM 詞對到多個時間詞就合併時間（start 取第一、end 取最後）指回該 span。② 兩邊先過同一套正規化。③ 保命網：對不齊的 clip 退整段，絕不錯標。④ v1 只對「詞數乾淨對上」的 clip 開逐字，含數字/縮寫的先退整段。

一眼版

雷後果怎麼避免對不齊時退路成本

⑦ 對抗驗證修正了兩個直覺

原本以為查證後
「Web Speech boundary 在 Safari/iOS 爛、Chrome 穩」 方向部分相反：MDN 把 Chrome/Android 標 partial（"does not fire as expected"），真正全壞的是 Android；Safari 會觸發但缺 charLength。淨結論不變：別把功能蓋在 boundary 上。
「msedge-tts 可靠地吐 WordBoundary」 機制與單位 CONFIRMED，但「可靠」被高估：README 範例一個 WordBoundary 都沒有、是逆向 API。→ build 時擷取並 commit sidecar，未來壞掉只壞 build。
「GPT-4o 換引擎時轉錄能補時間」 CONFIRMED：GPT-4o TTS 不吐 word timestamp，gpt-4o-transcribe 也給不了 word 級——只有 whisper-1 能。正是要解耦的理由。

⑧ 技術名詞解釋

TTS / STT
TTS＝文字→語音（我們現在做的）；STT/轉錄＝反方向語音→文字。類比：TTS 是念稿、STT 是聽寫。本案發音用 TTS，只有反推時間（Producer B）才用 STT/對齊。
WordBoundary metadata
TTS 引擎一邊念一邊回報「念到哪個詞、第幾毫秒到第幾毫秒」。類比：播音員旁邊有人即時按每個字的碼錶。本案＝Producer A 的免費時間來源。
timing manifest（<hash>.words.json / sidecar）
一份逐詞 start/end 秒數的小清單。類比：卡拉 OK 字幕時間軌。本案＝解耦的穩定介面。
forced alignment（強制對齊）
已有正確文字＋音檔，把每個字對齊到時間軸（不是猜字，是對時間）。類比：拿劇本標出每句台詞落在影片第幾秒。本案＝Producer B（WhisperX/MFA/aeneas）。
content-addressable hash（內容定址）
用內容的 SHA-256 當檔名/key，相同內容必得相同 hash。類比：用指紋當門牌。本案＝text→MP3→sidecar 全共用。
Web Speech API · boundary 事件 · charIndex
瀏覽器內建 TTS；念到詞邊界發 boundary，附「念到原文第幾字元」。本案＝fallback；手機不可靠，故只當桌面漸進增強。
rehype plugin / AST
Astro build 時把 Markdown/HTML 變成樹（AST）再改寫。類比：送印前自動在每個字外套標記。本案＝新 inject-word-spans.mjs 在 build 時把詞包 <span>。
tokenization / 斷詞漂移
把字串切成詞；漂移＝引擎切的詞 vs DOM 空白切的詞對不齊（數字、縮寫、連字號、彎引號）。本案最大實作風險，要序列對齊。
100-nanosecond units
Edge metadata 的時間單位，1 秒 = 1e7 tick。本案換算 秒 = Offset / 1e7。
SSML
給 TTS 的 XML 標記；&<> 要 escape。本案 escapeForSsml 已處理，但 escape 後有 WordBoundary desync bug。
timeupdate / requestAnimationFrame（rAF）
兩種播放中定時回呼。timeupdate 約 250ms 一次（太疏）；rAF 每幀 ~16ms（順）。本案 runtime 用 rAF。
解耦（decoupling）/ 穩定介面
把會變的東西藏在固定介面後，換實作不影響使用方。類比：插座規格不變、電器隨便換。本案＝sidecar 是插座，Edge/Whisper/GPT 是電器，runtime 不用動。
